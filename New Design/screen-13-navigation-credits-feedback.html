What this screen resolves

Navigation entry points

  • Home topbar trailing icons: Search · Bookmarks · ⋯ (overflow). Three icons, in that order, on the right side of the topbar.
  • Bookmarks gets a subtle dot indicator when there are unread/new bookmarks (e.g., user saved something on another device — moot for v1, but the affordance is there for v2).
  • Overflow menu opens as a 220 px popover anchored top-right, with three items: Settings · Send feedback · About this app. Backdrop scrim at 55% black. Tap scrim to dismiss.
  • Library screen keeps its inline search field at top of list. Tapping it routes to the full Search screen with field auto-focused. Same Search screen, two entry points (Home icon + Library inline field).
  • Chapter List, Verse List, Verse Detail do NOT get search/bookmarks icons. The user is in a reading flow; don't distract. Back-button only.
  • 3-tab bottom nav preserved. Today / Practice / Texts. Adding more tabs would signal shallow structure.

Credits & Attributions screen

  • Sūtra-style enumeration with Devanāgarī numerals. Each entry is numbered १, २, ३... in saffron. Mirrors how a classical Sanskrit śāstra enumerates its sources at the beginning of a commentary.
  • Section header "श्रद्धा · With gratitude" sets the tone. श्रद्धा (śraddhā) is the Sanskrit word for the kind of trust-with-respect that scholarly tradition is built on — not just "credits" as engineering attribution.
  • Lineage footer uses the palm-leaf binding-line motif and ends with the Bṛhadāraṇyaka Upaniṣad blessing "सर्वे भवन्तु सुखिनः" — every scriptural work historically ends with such a śānti mantra. This is one place where the heritage flourish is exactly right.
  • External-link arrows on rows that link out (GRETIL, sacred-texts.com, Project Madurai). Tools (libraries, fonts) don't link out — just metadata.
  • Sections grouped meaningfully: Vedas & Upaniṣads / Itihāsa-Purāṇa / Darśana texts / Tamil traditions / Tools used. Not alphabetical — by domain.
  • Voice: "Every verse in this app comes from a real academic source. Nothing is invented." This is contrarian copy that distinguishes the app from competitors. Keep this language.

Send Feedback screen

  • Two-step flow: pick kind → compose. Kind selection lets the message reach the right place in the backend (or filter inbox) without making the user type "this is a bug" themselves.
  • Four kinds: Bug report · Idea or suggestion · Text or translation error · Something else. The "Text error" kind is critical for a scripture app — scholars will report typos, and the app needs to make that easy.
  • Compose state: Selected kind appears as a saffron pill at the top, with version metadata on the right. Textarea takes most of the space. Two checkboxes at the bottom for device info attachment and reply consent.
  • "Attach device info" defaulted ON. Engineers need it. "Allow reply via email" defaulted OFF — user opts in. Privacy-respecting defaults.
  • Voice in the prose: "The app is built and maintained by one person — your reports and ideas shape what comes next." Honest, sets expectations, builds rapport. This is your competitive advantage as an indie developer; lean into it.
  • Send button disabled until message has content. Standard.

Settings rows now resolved

  • "Credits & attributions" row → routes to /settings/credits (Group 2 above)
  • "Privacy Policy" row → opens external URL via browser. No internal screen. (External-link arrow icon already shown in Settings spec.)
  • "Terms of Service" row → opens external URL via browser. Same pattern.
  • "Send feedback" row → routes to /settings/feedback (Group 3 above)
  • About this app (from overflow menu) → routes to the same Credits screen, or optionally a smaller About screen. Recommendation: same destination for v1.
